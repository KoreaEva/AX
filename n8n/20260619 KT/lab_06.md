> **"박 엔지니어, 장애 보고서 오늘 오후까지 정리해서 올려줘요. 보기 좋게."**

> 워드, 노션, 옵시디언에 붙여 넣으면 자동으로 서식이 적용됩니다.

## 실습 1 — 마크다운으로 장애 보고서 작성하기

> **배우는 것:** 마크다운을 활용하면 지루하고 반복적인 글쓰기 시간을 획기적으로 단축할 수 있습니다.

### 스토리

박준혁은 장애 처리 내용을 머릿속에 다 가지고 있습니다.  
그런데 이걸 보고서 형식으로 정리하려니 막막합니다.  
클로드에게 마크다운 형식으로 써달라고 해보기로 했습니다.

---

### ❌ 박준혁의 첫 번째 시도

```
어제 새벽에 기지국 장애 있었는데 보고서 써줘
```

**예상 결과:** 클로드가 뭔가를 쓰긴 했는데... 우리 팀 상황이 하나도 안 들어가 있습니다.

**박준혁:** "이게 아니잖아. 내 상황을 모르는데 어떻게 알겠어."

---

### ✅ 좋은 프롬프트 — 정보를 담아서 (직접 입력해보세요)

```
다음 내용을 마크다운으로 장애 보고서 형식으로 작성해줘.

[장애 정보]
- 발생 일시: 2026년 6월 22일 (일) 오전 02:15
- 복구 완료: 오전 03:40 (총 1시간 25분)
- 영향 범위: 수도권 북부 기지국 23개소
- 장애 원인: 코어 라우터 펌웨어 업데이트 후 라우팅 테이블 손상
- 조치 내용:
  1. 오전 02:15 모니터링 시스템 알람 수신
  2. 오전 02:30 현장 출동 및 원인 파악
  3. 오전 03:10 라우팅 테이블 롤백 조치
  4. 오전 03:40 전 기지국 정상 복구 확인
- 재발 방지: 펌웨어 업데이트 전 스테이징 환경 테스트 의무화

제목, 요약, 장애 현황, 조치 내용, 재발 방지 섹션으로 구성해줘.
```

---

### 실습 포인트

- [ ] 나쁜 프롬프트를 입력해서 결과를 확인한다
- [ ] 좋은 프롬프트를 입력해서 결과를 비교한다
- [ ] 결과물을 노션이나 워드에 붙여 넣어서 서식이 자동 적용되는지 확인한다

### 내가 받은 결과 메모

```
(여기에 클로드의 답변을 붙여넣으세요)
```

## 실습 2 — 맞춤 글쓰기 스타일 설정하기

> **배우는 것:** 클로드의 스타일 기능을 사용하면 매번 "전문적으로 써줘", "간결하게 써줘"를 반복하지 않아도 됩니다.

### 스토리

박준혁은 보고서를 두 가지 버전으로 써야 합니다.  
- **내부 팀용:** 기술 용어 그대로, 빠르게 핵심만  
- **경영진 보고용:** 쉬운 말로, 임팩트 중심으로  

매번 "쉽게 써줘", "전문적으로 써줘" 반복하기 귀찮습니다.  
스타일을 미리 만들어두기로 했습니다.

---

### 스타일 설정 방법

1. 클로드 입력창에서 `[+]` 버튼 클릭
2. `[스타일 사용]` 클릭 → 기본 스타일 5개 확인
3. `[스타일 생성 및 편집]` 클릭
4. `[맞춤 스타일 만들기]` 클릭
5. `[사용자 지정 지침 사용(고급)]` 선택 후 아래 지침 입력

---

### 스타일 A — 기술팀 내부용 (직접 만들어보세요)

**스타일 이름:** `기술 내부 보고`

**지침 입력 내용:**
```
- 기술 용어를 그대로 사용하세요 (IP, 라우터, 펌웨어 등 풀어 쓰지 마세요).
- 결론을 먼저 쓰고 근거를 나중에 씁니다.
- 번호 목록과 시간 순서를 중심으로 작성하세요.
- 불필요한 인사말이나 마무리 문장 없이 핵심만 담으세요.
- 200자 이내로 요약 가능한 경우 요약본을 먼저 제시하세요.
```

---

### 스타일 B — 경영진 보고용 (직접 만들어보세요)

**스타일 이름:** `경영진 보고`

**지침 입력 내용:**
```
- 기술 용어는 쉬운 말로 바꿔 설명하세요 (예: "라우팅 테이블" → "데이터 경로 정보").
- 비즈니스 영향(서비스 중단 시간, 영향 고객 수)을 앞에 배치하세요.
- 원인과 해결책을 각각 한 문장으로 요약하세요.
- 재발 방지 대책을 구체적으로 명시하세요.
- 전문 용어보다 임팩트와 결과 중심으로 서술하세요.
```

---

### 두 스타일 비교 실습

같은 장애 내용을 두 스타일로 각각 입력해서 비교해보세요.

**공통 프롬프트 (두 스타일에 각각 적용):**
```
어제 새벽 기지국 장애 보고서 한 단락으로 요약해줘.
장애 원인은 라우터 펌웨어 업데이트 후 라우팅 테이블 손상이고,
1시간 25분 만에 복구했어. 수도권 북부 23개 기지국 영향.
```

| 항목 | 기술 내부 보고 스타일 | 경영진 보고 스타일 |
|------|------------------|----------------|
| 첫 문장 | | |
| 기술 용어 사용 | | |
| 길이 느낌 | | |

---

### 실습 포인트

- [ ] 두 스타일을 직접 만들어본다
- [ ] 같은 프롬프트를 두 스타일에 각각 적용해서 결과를 비교한다
- [ ] 기본 제공 스타일(간결한, 격식있는 등)과도 비교해본다

## 실습 3 — 문서 첨부로 기존 자료 활용하기

> **배우는 것:** 클로드는 파일을 첨부해서 내용을 분석하고 사용할 수 있습니다.  
> 기존 문서를 수정하거나, 여러 자료를 종합해서 새로운 문서를 만들 수 있습니다.

### 스토리

박준혁의 팀에는 월간 장애 보고서 양식이 있습니다.  
매달 이 양식에 맞춰 내용을 채워야 하는데, 형식 맞추는 것 자체가 일입니다.  
기존 양식 파일을 클로드에게 주고, 이번 달 내용으로 채워달라고 해보기로 했습니다.

---

### 지원 파일 형식 확인

클로드에 첨부할 수 있는 파일:

- 문서: PDF, DOCX, TXT, RTF, 마크다운
- 스프레드시트: XLSX, CSV
- 프레젠테이션: PPTX
- 이미지: JPG, PNG, GIF, WebP (텍스트 추출 가능 — OCR 지원)
- 소스 코드: Python, JS, JSON, XML, HTML 등

---

### 실습 A — 기존 보고서 양식에 내용 채우기 (직접 해보세요)

**1단계:** 기존 보고서 파일(PDF 또는 DOCX)을 클로드 입력창에 드래그 앤 드롭

**2단계:** 아래 프롬프트 입력:
```
첨부한 파일은 우리 팀 월간 장애 보고서 양식이야.
이 양식 형식 그대로 유지하면서, 아래 이번 달 장애 내용으로 채워줘.

[6월 장애 내역]
1. 6월 3일: 부산 기지국 3개소 전원 공급 불안정 → 2시간 복구
2. 6월 15일: 대전 데이터센터 네트워크 스위치 교체 중 30분 단절
3. 6월 22일: 수도권 북부 23개소 라우팅 장애 → 1시간 25분 복구

마크다운 형식으로 출력해줘.
```

---

### 실습 B — 장애 로그 이미지에서 텍스트 추출하기 (직접 해보세요)

박준혁은 현장에서 화이트보드에 장애 타임라인을 적었습니다.  
사진을 찍어 클로드에 올리면 텍스트로 바꿔줍니다.

**프롬프트:**
```
이 사진에서 텍스트를 모두 추출해줘.
추출한 내용을 시간 순서대로 정리해서 장애 타임라인 표로 만들어줘.
```

> 손글씨, 화이트보드 사진, 명함 스캔도 모두 가능합니다.

---

### 파일 첨부 시 주의사항

박준혁이 꼭 기억할 것들:

- [ ] IP 주소, 내부 서버 접속 정보 등 민감 정보는 삭제 후 첨부
- [ ] 파일명을 `보고서1.pdf` 대신 `2026-06_장애보고서.pdf` 처럼 내용이 드러나게 저장
- [ ] 100페이지 매뉴얼 중 3페이지만 필요하면 해당 페이지만 추출해서 첨부
- [ ] 용량 초과 시 파일을 여러 개로 나누어 순서대로 업로드

---

### 실습 포인트

- [ ] 기존 보고서 양식 파일을 첨부해서 내용 채우기를 해본다
- [ ] 현장 메모 사진을 찍어 텍스트 추출을 해본다
- [ ] 파일명에 날짜와 내용을 넣는 습관을 들인다

## 실습 4 — 클로드와 질의응답하며 완성도 높은 문서 만들기

> **배우는 것:** 모델이 똑똑해질수록 프롬프트는 간단해도 됩니다.  
> 부족한 정보를 클로드가 스스로 물어보기 때문입니다.

### 스토리

박준혁은 이번 장애를 계기로 **재발 방지 제안서**를 써서 팀장에게 올리려고 합니다.  
뭘 써야 할지 구조가 안 잡힙니다.  
이번엔 클로드에게 먼저 물어봐달라고 해보기로 했습니다.

---

### 실습 프롬프트 (직접 입력해보세요)

```
네트워크 장애 재발 방지 제안서를 만들어줘.
더 필요한 정보가 있으면 나에게 질문해.
더 나은 결과물을 위해 필요한 맥락이 있다면 알려줘.
가능하다면 표나 체크리스트처럼 바로 활용할 수 있는 형태로 만들어줘.
```

---

### 예상되는 클로드의 질문들

클로드가 다음과 같은 것들을 물어볼 수 있습니다:

1. 어떤 장애가 발생했나요?
2. 제안서를 보는 대상이 누구인가요? (경영진 / 기술팀)
3. 문서 분량은 어느 정도로 원하나요?
4. 현재 운영 중인 모니터링 시스템이 있나요?

**박준혁의 답변 예시:**
```
1. 라우터 펌웨어 업데이트 후 라우팅 테이블 손상 → 기지국 23개소 1시간 25분 장애
2. 기술팀 팀장 + 경영진 동시 보고용
3. A4 2페이지 이내
4. 있음 — 나거스(Nagios) 기반 모니터링 운영 중
```

---

### 실습 포인트

- [ ] 위 프롬프트를 그대로 입력해서 클로드가 어떤 질문을 던지는지 확인한다
- [ ] 클로드의 질문에 박준혁의 상황에 맞게 답변해본다
- [ ] 대화 과정을 통해 완성된 제안서가 처음 예상보다 얼마나 구체적인지 비교한다
- [ ] 마지막으로 "마크다운 없이 일반 텍스트로 다시 출력해줘"도 시도해본다
